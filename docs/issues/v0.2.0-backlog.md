# v0.2.0 Issue Backlog

This file contains issue-ready drafts for the v0.2.0 milestone.

## 1. Add Dock Item Context Menu

Labels: `enhancement`, `ui`, `v0.2.0`

### Problem

Common Dock item actions currently depend heavily on edit mode and settings flows. A context menu would make direct item management easier.

### Scope

- Add right-click menu for Dock items.
- Include actions: open, edit, remove, change icon, open target location when applicable.
- Keep dangerous actions gated by confirmation where needed.
- Respect edit mode for mutating actions if the existing product behavior requires it.

### Acceptance Criteria

- Right-clicking a Dock item opens a context menu.
- Existing open behavior is unchanged.
- Remove and edit actions update config safely.
- Menu does not appear for empty Dock surface unless there is a clear action.

## 2. Add Broken Item State and Relink Flow

Labels: `enhancement`, `reliability`, `v0.2.0`

### Problem

If a target path is deleted or moved, the Dock item can fail without a clear recovery path.

### Scope

- Detect missing app/file/folder targets.
- Show a clear broken-item visual state.
- Add an action to relink the target.
- Preserve item metadata where possible.

### Acceptance Criteria

- Missing local paths are detected before launch.
- The user can relink a broken item.
- Relinked items save correctly.
- URL items are not incorrectly marked broken.

## 3. Validate and Repair Config on Load

Labels: `reliability`, `data`, `v0.2.0`

### Problem

Malformed config should not break startup or leave the UI in a partial state.

### Scope

- Validate required fields and default optional fields.
- Repair recoverable config problems.
- Fall back to backup config when the main config is unreadable.
- Log repair decisions.

### Acceptance Criteria

- App starts with missing optional fields.
- App handles invalid item entries without crashing.
- Backup-assisted recovery path is documented.
- Repair events are written to local logs.

## 4. Add Config Backup and Restore UX

Labels: `enhancement`, `data`, `v0.2.0`

### Problem

The app has local config and backup behavior, but users do not have a clear manual backup/restore workflow.

### Scope

- Add backup config action in settings.
- Add restore config action in settings.
- Validate restored config before applying.
- Keep file operations local and explicit.

### Acceptance Criteria

- User can export current config.
- User can restore a valid config file.
- Invalid restore input is rejected with a clear message.
- Restore does not erase the previous config without a backup.

## 5. Improve Log Access and Diagnostics

Labels: `reliability`, `developer-experience`, `v0.2.0`

### Problem

Errors are logged locally, but the troubleshooting path should be easier during development and user feedback.

### Scope

- Keep the existing open log folder action.
- Add a copy diagnostics summary action.
- Include app version, config path, log path, OS version, and basic runtime information.
- Avoid collecting private Dock item target paths in the summary unless the user explicitly chooses to share files.

### Acceptance Criteria

- User can copy a diagnostics summary from settings.
- Summary avoids sensitive item target data by default.
- Unexpected exceptions continue to be logged.

## 6. Record Startup Time and Update Performance Baseline

Labels: `performance`, `instrumentation`, `v0.2.0`

### Problem

Memory measurement is now repeatable, but startup timing and release-to-release comparison need a clearer habit.

### Scope

- Record startup duration in local logs.
- Document the measurement command for each release.
- Update `docs/15-performance-baseline.md` after v0.2.0 work.

### Acceptance Criteria

- Startup duration is recorded once per app launch.
- Performance baseline doc contains v0.1.1 and current measurements.
- Release checklist references the updated baseline.

## 7. Automate GitHub Release Packaging

Labels: `ci`, `release`, `v0.2.0`

### Problem

The project can build packages locally, but GitHub Releases are still manual.

### Scope

- Add a tag-triggered GitHub Actions workflow.
- Build installer and portable zip.
- Upload artifacts to the GitHub Release.
- Keep manual local release instructions as fallback.

### Acceptance Criteria

- Pushing a `v*` tag can produce release artifacts.
- Release workflow does not require secrets for unsigned builds.
- Failed release builds surface logs clearly.
